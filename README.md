# HART — Human Art Gallery on Solana

HART is a public art platform built around a single token (**hART**) on Solana. Artists mint, collectors collect, and a weekly community canvas burns 100% of its auction proceeds on-chain.

This repo is the **front-end + build pipeline**. Data and auth live in Supabase; settlement lives on Solana.

## Architecture in one diagram

```
   Browser  ──►  Static HTML/CSS/JS (this repo, served by Vercel)
                       │
                       ├──► Supabase (Postgres + RLS, anon key in browser)
                       │       └─ row-level security gates every read/write
                       │
                       └──► Solana RPC (wallet adapter)
                               └─ hART token + on-chain settlement
```

- No custom backend server. Edge functions in Supabase handle anything that needs `service_role`.
- The platform wallet is a **relay**, not custody — it batches burns into single auditable transactions.
- All burns go to a public burn address; anyone can verify them.

## Running locally

```bash
# 1. Install (only sets up the git pre-commit hook path)
npm install

# 2. Add a .env file with your Supabase anon key + platform wallet pubkey
#    (See SECURITY.md for the secret inventory.)
cp .env.example .env  # then fill in values

# 3. Build config.js from env, then serve the static site
npm run dev
```

`build.js` writes `js/config.js`. That file is gitignored — it's regenerated on every build, locally and on Vercel.

## Security

- All secrets policy: [`SECURITY.md`](SECURITY.md)
- The Supabase **anon** key is public by design (it reaches the browser). Data is protected by Row Level Security, not by hiding the key.
- The Supabase **service_role** key never touches this repo.
- Run `npm run security-check` before pushing. The pre-commit hook runs it automatically.

If you find a vulnerability, please open a private security advisory on GitHub rather than a public issue.

## Contributing

This is a small, opinionated codebase. Before opening a PR:

1. Read the relevant Bible chapter so you understand the intent of the area you're touching.
2. Keep changes scoped — one concern per PR.
3. No frameworks, no build step beyond `build.js`. Vanilla stays vanilla.
4. Don't introduce hardcoded keys, URLs, or addresses. Use `window.HART_CONFIG` / `window.HART_SUPABASE`.
5. `npm run security-check` must pass.
